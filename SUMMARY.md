# gRPC Header Kit — Solution Summary

> Internal solution report · Audience: leadership + engineers · From: Receiver Team

## TL;DR

Add a **routing-header layer auto-derived from the payload** on top of the existing 30 tx methods, so APISIX can route without parsing the payload. Consistency is guaranteed by "header and body share one source," not by manual sync.

## Problem

- Existing gRPC interface: **4 services / 30 tx methods**, payloads already generated.
- Missing a **header layer for APISIX context-based routing**; today routing requires parsing the payload — tight coupling, high cost.

## Constraints

- **gRPC-compliant**: metadata keys lowercase, values restricted to printable ASCII → values are always URL-encoded.
- **Metadata ≤ 8 KB** (hard limit).

## Core Mechanism

The mapping is annotated on proto fields via a custom option; headers are auto-generated from the **same message instance**.

```
proto field [(routing.header)]
        │   body = SSOT
        ▼
   one message instance
    ├─→ body     (existing payload, untouched)
    └─→ headers  (auto-generated by this kit, reads the same tree)
```

Because body and header read the **same message tree**, "body has it, header drops it" is structurally impossible. All other requirements follow from this mechanism:

- APISIX routes on headers only, never touching the payload (header = a projected subset of the payload)
- No hardcoding on interface version change (edit the proto, headers follow)
- Extensible (the option can add groups / fields, backward-compatible)

## Domain Context Mapping

| Header | Cardinality | Maps to |
|---|---|---|
| `tool-header` | 1 (required) | tool / chamber level |
| `lot-header` | 1..N | one per Lot |
| `mask-header` | 0..N | one per Mask |

- Format: URL-encoded `key=value&key=value`.
- Multiple entries = gRPC repeated metadata (same key, multiple values); no index suffixes.

## Cardinality × 8 KB: Verified

Worst case **1 tx = 25 lots + 3 masks**:

| Item | Value |
|---|---|
| Total (tool ×1 + lot ×25 + mask ×3) | ≈ 1.5 KB |
| 8 KB budget used | ~19% |
| Headroom | ~81% (6.6 KB) |
| Theoretical ceiling | ~127 lots |

Conclusion: enumerating headers per-item is safe at real cardinality; **no degradation strategy needed**.

## Interface — Three Parts

| Part | Content | Source |
|---|---|---|
| body | existing payload | sender (untouched) |
| headers | three routing headers | this kit, auto-generated from proto option |
| sample code | reflection (entry) / plugin codegen (production) | this kit |

- Plugin codegen pushes "forgot to update a header" detection to **compile time**.
- Verified end-to-end (protobuf 3.21.12): generated code compiles, output byte-identical; tagging a non-scalar field fails at build with a clear message.

## Governance

- Spec lives in a **governed repo**; `header_options.proto` = contract SSOT, versioned.
- Senders only `import`; they never copy the rules.
- Changing a group's semantics = breaking change → bump version + notify.

## Open (non-blocking)

- How APISIX consumes repeated headers (per-item vs comma-merged) — needs a live test; sender assumes "per-item repeated" for now.
- 8 KB is a conservative assumed limit; actual gateway setting TBC (does not affect the capacity conclusion above).
